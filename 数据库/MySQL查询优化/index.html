MySQL查询优化

2021-02-14 数据库 MySQL Comments Word Count: 2.6k(words)

查询优化

基于成本的优化

MySQL执行一个查询可以有不同的执行方案，它会选择其中成本最低的方案执行，这个成本最低的方案就是所谓的执行计划。这个成本主要包含I/O成本和CPU成本两方面。

对于InnoDB存储引擎，页是磁盘和内存交互的基本单位，MySQL的设计者规定默认值：

  • I/O成本：读取一个页的成本是1.0。
  • CPU成本：读取以及检测一条记录是否符合搜索条件的成本是0.1。

单表查询的成本

对于单表查询，查询优化器确定执行计划的流程如下：

  1. 根据搜索条件找出所有可能使用到的索引。
  2. 计算全表扫描的成本。
  3. 计算使用各种索引方案的成本。
  4. 选择各种方案中成本最低的一个作为执行计划。

对于成本计算，最重要的信息是Rows和Data_length。以上表的统计信息为例，我们可以得到该表有9693条记录，1589248Byte/(161024Byte)即97页。那么全表扫描的的成本就是I/O成本(`97.01.0+1.1=98.1)加上CPU成本(9693*0.1+1.0=1939.6`)等于1068.4。1.1和1.0是硬编码的微调常数。

注：全表扫描B+树非叶子节点是不需要被访问的，这里计算成本时忽略了这点。

索引统计数据

当使用非唯一二级索引和IN关键字时，IN子句中可能存在许多单点区间，例如：

由于一个单点区间可能存在多条匹配的记录，因此该索引方案的成本计算较为复杂。

  1. 当IN子句中单点区间较少时，一个单点区间中有多少相同记录可以通过index dive估算。index dive是指，计算B+树非叶子节点中，区间最左记录和区间最右记录之间一共有多少条记录，由于一条记录对应着B+树下一层的一页，因此就可以得出B+树叶子节点中区间占了多少页。接着沿着B+树叶子节点区间最左记录向右读10个页面，计算平均每个页面中包含多少记录，然后用这个平均值乘以区间最左记录和区间最右记录之间的页面数量就可以计算区间中总记录条数了。
  2. 而当IN子句中单点区间非常多时，通过index dive进行成本计算的损耗就相当大了。因此MySQL的设计者提供了一个系统变量eq_range_index_dive_limit，其默认值为200，当IN子句参数个数小于200时就使用index dive估算成本，大于200时则通过索引统计数据来计算成本了。

像会为每个表维护一份统计数据一样，MySQL也会为表中的每一个索引维护一份统计数据，查看某个表中索引的统计数据可以使用SHOW INDEX FROM 表名的语法。该统计数据中最重要的一项是索引列基数(Cardinality)，表示索引列不重复值的记录条数。

使用Rows/Cardinality可以计算得出平均一个值重复多少次，用该值乘以IN的参数个数就可得出大致需要回表的记录数。

注意：使用Cardinality计算单点区间对应的索引记录数虽然简单，但是其缺点就是不精确。如果使用IN子句时发现没有使用到索引，那可能就是因为IN的参数个数大于了200，导致成本计算结果甚至高于全表扫描的成本，这时就需要调高eq_range_index_dive_limit的值了。

连接查询的成本

Condition Filtering

为简单起见，只介绍两表连接的成本，它由以下两部分组成：

  1. 单次查询驱动表的成本。
  2. 多次查询被驱动表的成本。

我们把对驱动表进行查询后得到的记录条数称之为驱动表的扇出（fanout）。很显然驱动表的扇出值越小，对被驱动表的查询次数也就越少，连接查询的总成本也就越低。计算驱动表的扇出的过程就称为condition filtering。

两表连接的成本分析

连接查询总成本 = 单次访问驱动表的成本 + 驱动表扇出 x 单次访问被驱动表的成本。

  • 对于外连接而言，驱动表总是固定的，因此只需要分别为驱动表和被驱动表选择成本最低的访问方法 。
  • 对于内连接而言，驱动表与被驱动表的位置可以互换，因此需要计算两种情况的成本。

当我们书写连接语句时，应该尽量减少驱动表的扇出以及单次访问被驱动表的成本。比如我们可以通过在被驱动表的连接列上建立索引，或者使被驱动表的连接列是该表的主键/非唯一二级索引列来降低单次访问被驱动表的成本。

多表连接的成本分析

对于n张表的连接，其连接顺序有n!种，计算每一种连接顺序的成本是不现实的，因此MySQL的设计者提出了以下方案减少多种连接顺序成本的计算：

  1. MySQL在计算各种连接顺序的成本之前，会维护一个全局的变量，这个变量表示当前最小的连接查询成本。如果在分析某个连接顺序的成本时，该成本已经超过当前最小的连接查询成本，就停止分析。
  2. 如果连接表的个数小于系统变量optimizer_search_depth的值，那么就继续穷举分析每一种连接顺序的成本，否则只对与optimizer_search_depth值相同数量的表进行穷举分析。
  3. 使用启发式规则（就是根据以往经验指定的一些规则），凡是不满足这些规则的连接顺序都不进行分析。通过系统变量optimizer_prune_level决定是否使用启发式规则。

调节成本常数

使用SHOW TABLES FROM mysql LIKE '%cost'查看存储在mysql库中的成本常数表：

MySQL在server层进行连接管理、查询缓存、语法解析、查询优化等操作，在存储引擎层执行具体的数据存取操作。也就是说一条语句在server层中执行的成本是和它操作的表使用的存储引擎是没关系的，所以关于这些操作对应的成本常数就存储在了server_cost表中，而依赖于存储引擎的一些操作对应的成本常数就存储在了engine_cost表中。

mysql.server_cost

MySQL8版本该表中常数如下：

成本常数名称 默认值 描述
disk_temptable_create_cost 20 创建基于磁盘的临时表的成本，如果增大这个值的话会让优化器尽量少的创建基于磁盘的临时表。
disk_temptable_row_cost 0.5 向基于磁盘的临时表写入或读取一条记录的成本，如果增大这个值的话会让优化器尽量少的创建基于磁盘的临时表。
key_compare_cost 0.05 两条记录做比较操作的成本，多用在排序操作上，如果增大这个值的话会提升filesort的成本，让优化器可能更倾向于使用索引完成排序而不是filesort。
memory_temptable_create_cost 1 创建基于内存的临时表的成本，如果增大这个值的话会让优化器尽量少的创建基于内存的临时表。
memory_temptable_row_cost 0.1 向基于内存的临时表写入或读取一条记录的成本，如果增大这个值的话会让优化器尽量少的创建基于内存的临时表。
row_evaluate_cost 0.1 这个就是我们之前一直使用的检测一条记录是否符合搜索条件的成本，增大这个值可能让优化器更倾向于使用索引而不是直接全表扫描。

关于临时表： MySQL在执行诸如DISTINCT查询、分组查询、Union查询以及某些特殊条件下的排序查询都可能在内部先创建一个临时表，使用这个临时表来辅助完成查询（比如对于DISTINCT查询可以建一个带有UNIQUE索引的临时表，直接把需要去重的记录插入到这个临时表中，插入完成之后的记录就是结果集了）。在数据量大的情况下可能创建基于磁盘的临时表，也就是为该临时表使用MyISAM、InnoDB等存储引擎，在数据量不大时可能创建基于内存的临时表，也就是使用Memory存储引擎。

改变成本常数的SQL语句示例：

mysql.engine_cost

MySQL8版本该表中常数如下：

成本常数名称 默认值 描述
io_block_read_cost 1.0 从磁盘上读取一个块对应的成本。请注意我使用的是块，而不是页这个词儿。对于InnoDB存储引擎来说，一个页就是一个块，不过对于MyISAM存储引擎来说，默认是以4096字节作为一个块的。增大这个值会加重I/O成本，可能让优化器更倾向于选择使用索引执行查询而不是执行全表扫描。
memory_block_read_cost 0.25 与上一个参数类似，只不过衡量的是从内存中读取一个块对应的成本。
